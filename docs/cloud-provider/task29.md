# Task 29 - Adding an application level gateway

### Extend the private-subnet setup to:

- Allocate a managed public primary IPv4 address for the gateway.
- Install `apt-cacher-ng` on the gateway through Cloud-init.
- Bind the cache service only to the gateway's private interface.
- Delay creation of `intern` until the cache service is ready.
- Configure APT on `intern` to use the gateway proxy.
- Enable package updates and upgrades on `intern`.
- Install supplementary packages on `intern` as proof of working outbound package access.

The implementation extends the existing `PrivateSubnet` module. It was formatted and validated, but no plan, apply, SSH connection, or live proxy test was performed.

## Solution

### 1. Add the TLS provider requirement

The readiness hook uses a dedicated SSH key generated by Terraform. `Modules/PrivateSubnet/providers.tf` therefore contains:

```terraform
terraform {
  required_providers {
    hcloud = {
      source  = "hetznercloud/hcloud"
      version = "1.60.1"
    }
    tls = {
      source = "hashicorp/tls"
    }
  }
}
```

The child module inherits both configured providers from the root configuration.

### 2. Allocate a primary IPv4 address

The gateway's Internet-facing address is now managed explicitly in `Modules/PrivateSubnet/main.tf`:

```terraform
resource "hcloud_primary_ip" "gateway" {
  name          = "gateway-primary-ip"
  location      = var.location
  type          = "ipv4"
  assignee_type = "server"
  auto_delete   = false
}
```

The primary IP uses the same `nbg1` location as the gateway. `auto_delete = false` prevents deletion of the server from implicitly deleting an independently managed Terraform resource.

The gateway's public network block attaches this address and disables automatic public IPv6 allocation:

```terraform
  public_net {
    ipv4         = hcloud_primary_ip.gateway.id
    ipv4_enabled = true
    ipv6_enabled = false
  }
```

The module output now returns the managed primary IP directly:

```terraform
output "gateway_public_ip" {
  value       = hcloud_primary_ip.gateway.ip_address
  description = "Public IPv4 address used to SSH into the gateway"
}
```

### 3. Generate a dedicated readiness key

Terraform needs to connect to the gateway before it can mark the proxy service as ready. The module generates an independent key for this check:

```terraform
resource "tls_private_key" "service_ready" {
  algorithm = "ED25519"
}
```

The gateway's Cloud-init input combines the user's ordinary SSH keys with the readiness public key:

```terraform
  user_data = templatefile("${path.module}/scripts/cloud-init-gateway.yml", {
    hostname     = "gateway"
    fqdn         = local.gateway_fqdn
    gateway_fqdn = local.gateway_fqdn
    gateway_ip   = local.gateway_private_ip
    intern_fqdn  = local.intern_fqdn
    intern_ip    = local.intern_private_ip
    ssh_public_keys = concat(
      values(var.ssh_public_keys),
      [tls_private_key.service_ready.public_key_openssh]
    )
  })
```

The corresponding private key is used only by Terraform's readiness connection. It is stored as sensitive Terraform state and is not written into the project directory.

### 4. Install apt-cacher-ng on the gateway

The gateway now uses the separate `Modules/PrivateSubnet/scripts/cloud-init-gateway.yml` template. Package updates are enabled and the proxy plus a readiness-test client are installed:

```yaml
package_update: true
package_upgrade: true
ssh_pwauth: false
disable_root: true
manage_etc_hosts: true
hostname: ${hostname}
fqdn: ${fqdn}

packages:
  - apt-cacher-ng
  - curl
```

The template retains the hardened `devops` user and local hosts template introduced in Task 28.

### 5. Bind the proxy to the private interface

The gateway's `runcmd` section modifies the package-provided configuration after installation:

```yaml
runcmd:
  - |
    if grep -q '^BindAddress:' /etc/apt-cacher-ng/acng.conf; then
      sed -i 's/^BindAddress:.*/BindAddress: ${gateway_ip}/' /etc/apt-cacher-ng/acng.conf
    else
      echo 'BindAddress: ${gateway_ip}' >> /etc/apt-cacher-ng/acng.conf
    fi
  - systemctl enable apt-cacher-ng
  - systemctl restart apt-cacher-ng
  - systemctl restart ssh || systemctl restart sshd
```

For the configured subnet, Terraform renders `${gateway_ip}` as `10.0.1.2`. `apt-cacher-ng` therefore listens on the internal network interface rather than the public primary IP.

The service uses its standard TCP port `3142`:

```text
http://10.0.1.2:3142
```

The gateway firewall exposes that port only to clients in the private subnet:

```terraform
  rule {
    direction  = "in"
    protocol   = "tcp"
    port       = "3142"
    source_ips = [var.private_subnet.ipAndNetmask]
  }
```

The proxy therefore remains unreachable through the gateway's public primary IP.

### 6. Add the service-ready dependency hook

The built-in `terraform_data` resource creates a dependency point without creating another cloud resource:

```terraform
resource "terraform_data" "apt_cacher_ready" {
  triggers_replace = [hcloud_server.gateway.id]

  connection {
    type        = "ssh"
    host        = hcloud_primary_ip.gateway.ip_address
    user        = "devops"
    private_key = tls_private_key.service_ready.private_key_openssh
    timeout     = "15m"
  }

  provisioner "remote-exec" {
    inline = [
      "until systemctl is-active --quiet apt-cacher-ng && curl --fail --silent --output /dev/null http://${local.gateway_private_ip}:3142/acng-report.html; do sleep 5; done"
    ]
  }
}
```

The readiness hook:

1. Waits for SSH to become available on the managed primary IP.
2. Checks that the `apt-cacher-ng` systemd service is active.
3. Requests the cache report through the private address `10.0.1.2:3142`.
4. Repeats every five seconds until both checks succeed.
5. Fails after the SSH connection's 15-minute timeout if the gateway never becomes ready.

`triggers_replace` reruns the check whenever Terraform replaces the gateway server.

### 7. Delay intern until the proxy is ready

The existing `intern` server now depends on the readiness resource as well as the subnet:

```terraform
resource "hcloud_server" "intern" {
  depends_on = [hcloud_network_subnet.private, terraform_data.apt_cacher_ready]
```

Terraform cannot begin creating `intern` until the remote readiness command completes successfully. This prevents its first package update from racing gateway initialization.

The template input supplies the private proxy URL:

```terraform
  user_data = templatefile("${path.module}/scripts/cloud-init.yml", {
    hostname        = "intern"
    fqdn            = local.intern_fqdn
    gateway_fqdn    = local.gateway_fqdn
    gateway_ip      = local.gateway_private_ip
    intern_fqdn     = local.intern_fqdn
    intern_ip       = local.intern_private_ip
    apt_proxy       = "http://${local.gateway_private_ip}:3142"
    ssh_public_keys = values(var.ssh_public_keys)
  })
```

### 8. Configure APT on intern

The `intern` Cloud-init template enables package updates and upgrades:

```yaml
package_update: true
package_upgrade: true
ssh_pwauth: false
disable_root: true
manage_etc_hosts: true
hostname: ${hostname}
fqdn: ${fqdn}
```

The `apt` configuration directs HTTP repository traffic through the application-level gateway:

```yaml
apt:
  preserve_sources_list: true
  conf: |
    Acquire::http::Proxy "${apt_proxy}";
```

Terraform renders this setting as:

```text
Acquire::http::Proxy "http://10.0.1.2:3142";
```

The private host still has no direct Internet route. APT reaches HTTP repositories through `apt-cacher-ng` on the dual-homed gateway.

### 9. Install proof packages on intern

The template declares three supplementary packages:

```yaml
packages:
  - curl
  - jq
  - plocate
```

Successful creation of `intern` and completion of its Cloud-init package stage provide evidence that package metadata and package downloads can traverse the gateway proxy.

## Dependency order

Terraform derives the following creation sequence:

```text
private network and subnet
├── managed gateway primary IPv4
├── readiness SSH key
└── gateway server
    └── Cloud-init installs and starts apt-cacher-ng
        └── terraform_data.apt_cacher_ready succeeds
            └── intern server is created
                └── Cloud-init updates APT and installs proof packages
```

The readiness hook is intentionally narrow. It does not configure the gateway; Cloud-init remains responsible for configuration, while Terraform only observes that the service is ready before releasing the dependent host.

When Task 29 was initially implemented, only static Terraform validation was performed:

```bash
terraform validate
```

Result:

## Verification after applying

The following commands were verified against the deployed infrastructure.

### Verify the primary IP

```bash
terraform output private_subnet
```

Output:

```text
gateway_public_ip = "88.99.224.53"
gateway_private_ip = "10.0.1.2"
intern_private_ip = "10.0.1.3"
```

### Verify the gateway service

```bash
ssh devops@88.99.224.53
sudo systemctl status apt-cacher-ng --no-pager
sudo ss -ltnp | grep 3142
curl -I http://10.0.1.2:3142/acng-report.html
```

Output:

```text
Active: active (running)
LISTEN ... 10.0.1.2:3142 ... apt-cacher-ng
HTTP/1.1 200 OK
```

The listener output must show `10.0.1.2:3142`, not `0.0.0.0:3142` or the public primary address.

### Verify intern through chained SSH

```bash
ssh -J devops@88.99.224.53 devops@10.0.1.3
```

Inspect the rendered proxy setting:

```bash
apt-config dump | grep -i 'Acquire::http::Proxy'
```

Expected output:

```text
Acquire::http::Proxy "http://10.0.1.2:3142";
```

Verify the supplementary packages:

```bash
dpkg-query -W curl jq plocate
```

Output:


## Result

The gateway now uses a managed primary IPv4 for Internet access and runs `apt-cacher-ng` exclusively on its `10.0.1.2` private interface. A Terraform readiness hook delays creation of `intern` until that service responds. The private host configures APT to use the gateway, enables updates and upgrades, and installs `curl`, `jq`, and `plocate` without receiving a public network interface.
